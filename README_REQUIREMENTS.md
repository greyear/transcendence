# ft_transcendence README Requirements

**Source**: ft_transcendence Project Specification v19.0, Chapter VI (Readme Requirements), Pages 27–29

This document outlines all mandatory sections and content requirements for the README.md file as specified in the official project documentation.

---

## Mandatory Base Requirements

### 1. Opening Line (REQUIRED)

**Format**: Italicised text on the first line

**Content**: Must include project attribution to 42 curriculum and all team member logins

**Example**:
```markdown
*This project has been created as part of the 42 curriculum by <login1>[, <login2>[, <login3>[...]]]*
```

**Status**: ✅ MUST BE INCLUDED

---

### 2. Description Section (REQUIRED)

**Purpose**: Allow anyone unfamiliar with the project to quickly understand what it is about

**Minimum Content**:
- Clear project name
- Project goal and objective
- Brief overview of functionality
- Key features list

**Additional Details** (per spec):
- Project description should clearly present what the project is
- Include goal and brief overview

**Status**: ✅ MUST BE INCLUDED

---

### 3. Instructions Section (REQUIRED)

**Purpose**: Provide information about compilation, installation, and/or execution

**Minimum Content**:
- All required prerequisites (software, tools, versions)
- Configuration instructions (e.g., `.env` setup)
- Step-by-step instructions to run the project
- How to build/deploy the application

**Status**: ✅ MUST BE INCLUDED

---

### 4. Resources Section (REQUIRED)

**Purpose**: List references and document AI usage

**Minimum Content**:
- Classic references related to the project topic (documentation, articles, tutorials)
- **MUST INCLUDE**: Description of how AI was used, specifying:
  - For which tasks AI was utilised
  - Which parts of the project benefited from AI assistance

**Status**: ✅ MUST BE INCLUDED

---

## Additional Mandatory Sections (for this project)

### 5. Team Information (REQUIRED)

**Purpose**: Document team structure, roles, and responsibilities

**For each team member mentioned in the opening line, provide**:
- Assigned role(s) (e.g., PO, PM, Tech Lead, Developer)
- Brief description of their responsibilities

**Format**: Can use table or list format

**Status**: ✅ MUST BE INCLUDED

---

### 6. Project Management (REQUIRED)

**Purpose**: Explain how the team organised and coordinated work

**Minimum Content**:
- How the team organised the work (task distribution, meeting schedule, etc.)
- Project management tools used (e.g., GitHub Issues, Trello, Jira)
- Communication channels used (e.g., Discord, Slack)

**Status**: ✅ MUST BE INCLUDED

---

### 7. Technical Stack (REQUIRED)

**Purpose**: Document all technology choices and provide justification

**Minimum Content**:
- Frontend technologies and frameworks used
- Backend technologies and frameworks used
- Database system(s) and justification for selection
- Any other significant technologies or libraries
- Justification for major technical choices

**Status**: ✅ MUST BE INCLUDED

---

### 8. Database Schema (REQUIRED)

**Purpose**: Document the data structure

**Minimum Content**:
- Visual representation OR description of the database structure
- Tables/collections and their relationships
- Key fields and data types

**Status**: ✅ MUST BE INCLUDED

---

### 9. Features List (REQUIRED)

**Purpose**: Provide complete inventory of implemented functionality

**Minimum Content**:
- Complete list of all implemented features
- Which team member(s) worked on each feature
- Brief description of each feature's functionality

**Status**: ✅ MUST BE INCLUDED

---

### 10. Modules (REQUIRED)

**Purpose**: Document chosen modules and their implementation

**Minimum Content**:
- List of all chosen modules (Major and Minor)
- Point calculation (Major = 2 pts, Minor = 1 pt)
- Justification for each module choice (especially for custom "Modules of choice")
- How each module was implemented
- Which team member(s) worked on each module

**Status**: ✅ MUST BE INCLUDED

---

### 11. Individual Contributions (REQUIRED)

**Purpose**: Clearly document each team member's work

**Minimum Content**:
- Detailed breakdown of what each team member contributed
- Specific features, modules, or components implemented by each person
- Any challenges faced and how they were overcome

**Status**: ✅ MUST BE INCLUDED

---

## Optional Content

The specification states: "Any other useful or relevant information is welcome"

**Examples**:
- Usage documentation
- Known limitations
- Future enhancements
- License information
- Credits and acknowledgements
- Deployment instructions
- API documentation
- Testing procedures

**Status**: ✅ RECOMMENDED (can positively impact evaluation)

---

## Additional Guidance

### Quality Standards

The README.md is described as "a critical part of your project evaluation" and should be:

- **Clear and well-organised** — Easy to navigate with logical section ordering
- **Complete with all required sections** — No missing mandatory content
- **Professional and easy to read** — Proper formatting, grammar, spelling
- **Honest about contributions and challenges** — Transparent breakdown of team efforts

### Consequences of Non-Compliance

> "A poor or incomplete README can negatively impact your evaluation."

---

## Checklist for README Completion

- [ ] **Opening Line**: Italicised, includes all team member logins
- [ ] **Description**: Project name, goal, overview, key features
- [ ] **Instructions**: Prerequisites, configuration, step-by-step execution steps
- [ ] **Resources**: Documentation references + AI usage description
- [ ] **Team Information**: Each member's role and responsibilities
- [ ] **Project Management**: Team organisation, tools, communication channels
- [ ] **Technical Stack**: Frontend, backend, databases, justification
- [ ] **Database Schema**: Structure, relationships, key fields, data types
- [ ] **Features List**: All features listed with owners and descriptions
- [ ] **Modules**: Module list, point calculation, justification, implementation details, owners
- [ ] **Individual Contributions**: Detailed breakdown of each member's work and challenges
- [ ] **Formatting & Quality**: Professional, clear, well-organised, no grammatical errors
- [ ] **Completeness**: All required sections present, no placeholders or empty sections
